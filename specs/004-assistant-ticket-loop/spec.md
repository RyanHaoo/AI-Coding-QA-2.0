## Clarifications

### Session 2026-07-03

- Q: 图片是否需要独立图片理解链路？ → A: 图片和文本一同由 Agent 多模态理解，不建立独立图片理解链路。

## 用户场景与验收 *(mandatory)*

### 用户故事 1 - 通过助手创建工单（优先级: P1）

质检员或管理员进入智能助手页面后，可以用自然语言描述现场质量问题，并可附带现场图片。助手需要在同一轮多模态输入中理解文本和图片，识别建单意图，追问缺失的必填信息，并在信息足够后展示一张待确认工单卡片。用户确认责任人、严重程度、专业类型、问题内容和图片后提交，系统创建工单并在对话中返回工单编号和可继续查看的成功反馈。

**优先级理由**: 助手建单是阶段 4 的最小可演示闭环，直接把自然语言输入连接到阶段 2 和阶段 3 已完成的工单业务对象。

**独立验收**: 使用质检员或管理员身份进入助手页面，输入一段包含位置和问题描述的话术，上传现场图片，确认 Agent 在同一轮多模态输入中结合文本和图片整理草稿，完成缺失信息追问、卡片确认和提交，确认新工单能在对应列表或详情中查到，且详情展示随工单保存的现场图片。

**验收场景**:

1. **Given** 当前身份为质检员且已选择项目，**When** 用户输入“C栋B户型采光系数低于标准，帮我创建工单”，**Then** 助手展示建单草稿卡片，并允许用户补全或调整严重程度、专业类型、责任人和问题详情。
2. **Given** 建单草稿缺少责任人，**When** 用户尝试提交，**Then** 助手必须提示选择当前项目内施工方责任人，而不是创建不完整工单。
3. **Given** 用户已在助手输入区上传现场图片，**When** 助手生成建单草稿，**Then** 草稿卡片或输入区展示图片预览，并允许用户在提交前删除不需要的图片。
4. **Given** 用户上传了现场图片但文字描述较少，**When** 助手整理建单草稿，**Then** Agent 在同一轮多模态输入中结合文本和图片生成问题摘要、位置线索或待补充提示。
5. **Given** 建单草稿信息完整且包含现场图片，**When** 用户确认提交，**Then** 助手回复创建成功、展示新工单编号，并且该工单出现在发起人的工单列表和责任人的待处理范围内，详情现场图片区展示已保存图片。
6. **Given** 当前身份为施工方，**When** 用户要求助手创建工单，**Then** 助手拒绝建单并说明只有质检员或管理员可以发起工单。

---

### 用户故事 2 - 通过助手查询有权工单（优先级: P2）

用户可以在智能助手中用工单编号、关键词、位置、状态表达或“我的工单”等自然语言查询工单。助手必须只返回当前身份有权查看的工单摘要，并提供进一步查看工单详情的入口或明确下一步。

**优先级理由**: 查单让助手从单向建单扩展为工单入口，支撑用户快速定位阶段 2 和阶段 3 中已有的业务数据；它依赖既有工单可见范围，但不依赖 P1 新建工单。

**独立验收**: 分别使用质检员、施工方和管理员身份输入编号、关键词和“我的待处理工单”等查询表达，确认助手只返回符合当前项目和身份范围的结果。

**验收场景**:

1. **Given** 当前身份有权查看工单 #23，**When** 用户输入“查询工单23”，**Then** 助手返回该工单的编号、状态、严重程度、位置、当前责任人和问题摘要。
2. **Given** 当前身份为施工方，**When** 用户输入“我的待处理工单”，**Then** 助手返回当前项目内当前责任人为自己的待处理工单摘要。
3. **Given** 当前身份为管理员，**When** 用户输入某个当前项目关键词，**Then** 助手可以返回当前项目内匹配的多张工单摘要。
4. **Given** 查询命中其他项目或无权工单，**When** 用户输入对应编号或关键词，**Then** 助手不得泄露工单详情，只能提示未找到可访问结果。

### 用户故事 3 - 呈现可演示的聊天体验（优先级: P3）

智能助手页面需要参考 Stitch 聊天界面，提供清晰的消息流、用户消息气泡、助手文本回复、建单草稿卡片、图片预览、提交成功反馈和底部输入区。界面应适合桌面和移动演示，用户能看出当前是在对话中完成建单或查单，而不是填写传统表单。

**优先级理由**: 聊天体验是原型指定的展示方式，影响演示理解度；但它服务于 P1/P2 的业务闭环，可在核心建单和查单规则明确后打磨。

**独立验收**: 打开助手页面后，确认消息流、输入区、建单草稿卡片和提交成功反馈在桌面和移动宽度下都清晰可读，并能完成至少一次建单或查单演示。

**验收场景**:

1. **Given** 用户进入助手页面，**When** 页面加载完成，**Then** 系统展示聊天消息区域和固定或易访问的输入区域。
2. **Given** 用户发送消息，**When** 助手生成回复，**Then** 用户消息和助手消息在视觉上可区分，且时间顺序清晰。
3. **Given** 助手识别到建单意图，**When** 信息足够或追问完成，**Then** 页面展示建单草稿卡片，卡片包含必填标识、可调整字段、图片预览和提交动作。
4. **Given** 工单创建成功，**When** 对话继续展示，**Then** 成功反馈与草稿状态可区分，用户能看到新工单编号。

### 边界情况

- 当用户输入为空或只有无意义文本时，助手必须提示用户输入要创建或查询的工单信息。
- 当建单缺少问题描述、详细位置或责任人时，助手必须追问缺失项或阻止提交并指出需要补全的内容。
- 当责任人不属于当前项目施工方时，助手必须拒绝提交并要求重新选择有效责任人。
- 当图片上传失败、格式不可用、多模态理解失败或用户删除图片时，助手必须明确展示图片当前状态，并允许用户继续无图建单、手动补充文字信息或重新选择图片。
- 当用户上传多张现场图片时，助手不得在阶段 4 增加额外数量上限；若现有上传能力或环境限制导致失败，必须逐项给出可理解反馈。
- 当当前身份缺失、已失效或没有项目上下文时，助手必须阻止建单和查单，并引导用户重新登录或选择身份。
- 当查询没有匹配工单时，助手必须展示无结果反馈，并提示可尝试编号、关键词或状态表达。
- 当用户要求知识问答、规范条文解释或超出阶段 4 范围的问题时，助手必须说明当前阶段暂不支持该能力，不得编造答案。
- 当工单创建或查询失败时，助手必须保留用户已输入或已整理的信息，并展示可理解的失败反馈。

## 需求 *(mandatory)*

### 功能需求

- **FR-001**: 系统必须提供智能助手页面，包含消息流、文本输入区、图片上传入口、用户消息、助手回复和可交互的建单草稿卡片。
- **FR-002**: 助手必须能识别自然语言中的建单意图和查单意图；无法识别时必须给出可继续输入的提示。
- **FR-003**: 只有质检员和管理员可以通过助手创建工单；施工方请求创建工单时必须被拒绝并说明原因。
- **FR-004**: 助手建单必须收集问题描述、详细位置、严重程度、专业类型和当前项目施工方责任人。
- **FR-005**: 当建单信息缺少问题描述、详细位置或责任人时，助手必须基于同一轮多模态输入中的文本和图片追问缺失信息，或在草稿卡片中标明必填项。
- **FR-006**: 助手必须展示待确认工单卡片，卡片至少包含问题描述、详细位置、严重程度、专业类型、责任人、问题详情和确认提交动作。
- **FR-007**: 用户必须能够在提交前调整建单草稿中的严重程度、专业类型、责任人、可选问题详情和现场图片。
- **FR-008**: 责任人只能从当前项目内施工方身份中选择。
- **FR-009**: 用户确认提交后，系统必须创建一张待处理工单，发起人为当前身份，当前责任人为所选施工方，并保留助手整理出的工单信息和用户确认保留的现场图片。
- **FR-010**: 工单创建成功后，助手必须返回新工单编号、创建成功状态和进一步查看该工单的明确入口或说明。
- **FR-011**: 新建工单必须进入已有工单可见范围：发起人可在自己的工单中看到，责任施工方可在自己的待处理工单中看到，管理员可在当前项目范围内看到。
- **FR-011a**: 新建工单包含现场图片时，图片必须随工单保存，并在工单详情的现场图片区域可见。
- **FR-011b**: 用户上传现场图片时，Agent 必须在同一轮多模态输入中结合文本和图片理解可见质量问题。
- **FR-011c**: 阶段 4 不得建立独立图片理解链路；图片不得形成独立字段、独立状态或特殊审核路径，最终工单内容仍以用户确认的草稿为准。
- **FR-011d**: 助手建单不得设置独立的现场图片数量上限，必须沿用已有工单图片上传能力和失败反馈。
- **FR-012**: 助手必须支持按工单编号查询当前身份有权访问的工单。
- **FR-013**: 助手必须支持按关键词、详细位置、问题描述摘要或状态表达查询当前身份有权访问的工单。
- **FR-014**: 助手必须支持“我的工单”“我的待处理工单”等当前身份范围查询。
- **FR-015**: 查单结果必须展示工单编号、状态、严重程度、详细位置、当前责任人、问题摘要和可继续查看详情的入口或说明。
- **FR-016**: 查单结果必须遵守当前项目和当前身份可见范围；无权工单不得展示详情。
- **FR-017**: 助手必须在无结果、无权限、身份缺失、必填信息缺失、责任人无效和提交失败时展示明确反馈。
- **FR-018**: 助手页面必须参考 Stitch 聊天界面的信息结构：用户消息气泡、助手文本回复、建单草稿卡片、图片预览、提交确认状态和底部输入区域。
- **FR-019**: 助手消息流必须保持本次页面会话中的上下文，使用户能在追问后继续补充建单信息或继续查询。
- **FR-020**: 本阶段不得实现 Coze/扣子知识接口、规范条文问答、语音输入或生产级多轮智能体能力；图片上传、预览、删除、多模态建单理解和随工单保存属于本阶段建单闭环。

### 关键实体 *(涉及数据时填写)*

- **助手会话**: 表示用户在助手页面的一次对话过程，包含用户消息、助手回复、追问状态、建单草稿和查单结果。
- **用户消息**: 表示用户输入的自然语言内容，可触发建单、补充建单信息、查询工单或无法识别反馈。
- **助手回复**: 表示助手对用户输入的可见响应，包括文本说明、追问、无权限反馈、无结果反馈、查单摘要或建单成功反馈。
- **建单草稿**: 表示助手从对话中整理出的待确认工单信息，包含问题描述、详细位置、严重程度、专业类型、责任人、问题详情和现场图片。
- **现场图片**: 表示用户在助手建单过程中上传并确认保留的图片资料，与文本一起作为同一轮 Agent 多模态输入，并随新工单保存到详情中展示。
- **多模态建单输入**: 表示用户文本和现场图片共同组成的一次 Agent 输入；系统不把图片拆成独立理解链路或独立业务数据。
- **查单结果**: 表示当前身份有权访问的工单摘要集合，用于在对话中快速定位目标工单。
- **当前身份可见范围**: 表示由当前用户、当前项目和当前角色共同决定的建单权限、责任人候选范围和查单结果范围。

## 成功标准 *(mandatory)*

### 可衡量结果

- **SC-001**: 演示者可在 5 分钟内用质检员身份完成“自然语言描述问题 -> 上传现场图片 -> 多模态输入生成草稿 -> 补全信息 -> 确认草稿 -> 创建工单 -> 在列表或详情中查到新工单和现场图片”的完整路径。
- **SC-002**: 演示者可在 5 分钟内用管理员身份完成同样的助手建单路径，并验证新工单归属当前项目。
- **SC-003**: 施工方请求创建工单时，系统在 1 次助手回复内明确拒绝，并说明当前身份不能发起工单。
- **SC-004**: 用户分别用编号、关键词和“我的待处理工单”三类表达查询时，助手能返回当前身份有权访问的结果或明确无结果反馈。
- **SC-005**: 查询其他项目或无权工单时，助手不会展示工单详情，人工验收可确认无敏感信息泄露。
- **SC-006**: 建单草稿卡片和图片预览在桌面和移动演示宽度下均可完整阅读和操作，用户消息、助手消息和成功反馈可清晰区分。
- **SC-007**: 阶段 4 交付前完成适用静态检查；如果无法运行，必须明确记录原因和风险。

## 假设

- 阶段 4 复用阶段 1 的登录身份、项目和角色边界，阶段 2 的工单数据底座，以及阶段 3 的工单状态、责任人规则和工单图片上传/保存能力。
- 助手创建的工单默认状态为待处理，当前责任人必须是当前项目内施工方。
- 助手的自然语言能力在本阶段以演示核心话术为准，不承诺覆盖任意复杂表达或长上下文推理。
- 聊天记录只需要支撑当前页面会话演示，不承诺跨设备、跨登录或长期历史保存。
- 建单草稿中的严重程度和专业类型可以根据用户表达预填；无法判断时使用可见默认值并允许用户在卡片中调整。
- 本阶段不接入 Coze/扣子知识接口；规范条文解释和知识问答进入阶段 6。图片上传、预览、删除、多模态建单理解和图片随工单保存已纳入阶段 4。
