# Quickstart: 工单状态流转与操作记录

## 前置条件

- 已完成阶段 1 和阶段 2。
- `.env.local` 已配置 Supabase URL、publishable key 和服务端 `SECRET_KEY`。
- 阶段 1 测试账号可登录。
- 阶段 2 工单 seed 已准备至少 12 张演示工单。
- 阶段 3 如新增 migration 或 Storage 配置，已应用到当前 Supabase 项目。

## 启动

```powershell
npm run dev
```

打开本地开发地址后，使用 `progress.md` 中的测试账号登录。建议优先使用：

- `wang.builder@example.com`: 施工方路径，验证解决、拒绝和指派。
- `li.qc@example.com`: 质检员路径，验证编辑和重新打开。
- `chen.admin@example.com`: 管理员路径，验证项目内管理干预。
- `ryan.multi@example.com`: 多身份路径，验证不同项目和角色边界。

## 验收路径 1: 解决待处理工单

1. 使用施工方身份登录。
2. 打开工单列表，确认默认筛选为待处理。
3. 打开一张当前责任人为自己的待处理工单。
4. 点击“解决”。
5. 不填写问题归因直接提交，预期系统提示补全。
6. 填写问题归因，可选填写预防建议，提交。
7. 预期详情状态变为已完成，处理信息展示问题归因，处理记录新增“解决”。
8. 返回待处理列表，预期该工单不再出现在待处理筛选中。
9. 切换到已结束或全部，预期可以看到该工单。

## 验收路径 2: 拒绝待处理工单

1. 使用施工方身份打开另一张待处理工单。
2. 点击“拒绝”。
3. 不填写拒绝原因直接提交，预期系统提示补全。
4. 填写拒绝原因并提交。
5. 预期详情状态变为已拒绝，处理记录新增“拒绝”并展示拒绝原因。

## 验收路径 3: 编辑问题信息和现场图片

1. 使用工单发起人、当前责任人或管理员身份打开待处理工单。
2. 点击“编辑”。
3. 修改严重程度、专业类型、问题描述、详细位置或问题详情。
4. 新增一张现场图片，并删除一张已有现场图片。
5. 点击取消，预期详情恢复编辑前内容。
6. 再次进入编辑，重复修改并点击保存。
7. 预期详情展示新问题信息和图片列表，处理记录新增“编辑”。
8. 预期工单编号、状态、创建时间、发起人、所属项目、当前责任人、问题归因、预防建议和处理记录在编辑模式中不可直接修改。

## 验收路径 4: 指派他人

1. 使用当前责任人或管理员身份打开待处理工单。
2. 点击“指派他人”。
3. 确认候选人只包含当前项目内另一名施工方。
4. 不填写指派原因直接提交，预期系统提示补全。
5. 选择施工方并填写指派原因后提交。
6. 预期详情中当前责任人变为新施工方，状态仍为待处理，处理记录新增“指派他人”。
7. 使用原责任人身份刷新工单列表，预期该工单不再出现在其待处理列表。
8. 使用新责任人身份登录，预期该工单出现在其待处理列表。

## 验收路径 5: 重新打开已结束工单

1. 使用工单发起人或管理员身份打开已完成或已拒绝工单。
2. 点击“重新打开”。
3. 不填写重新打开原因直接提交，预期系统提示补全。
4. 填写原因并提交。
5. 预期工单状态恢复为待处理，当前责任人保持不变。
6. 预期处理记录新增“重新打开”，历史拒绝、解决或编辑记录仍可查看。
7. 使用当前责任人身份打开待处理列表，预期该工单重新出现。

## 验收路径 6: 角色和状态可见性

分别使用质检员、施工方和管理员身份查看待处理、已完成和已拒绝工单：

- 待处理工单的发起人、当前责任人和管理员可看到“编辑”。
- 待处理工单的当前责任人和管理员可看到“解决”“拒绝”“指派他人”。
- 已完成和已拒绝工单的发起人和管理员可看到“重新打开”。
- 已完成和已拒绝工单不展示“编辑”“解决”“拒绝”“指派他人”。
- 无权身份不展示操作入口。

## 异常验收

- 直接提交无权操作时，系统拒绝并展示明确反馈。
- 工单状态已变化时，旧页面提交被拒绝并提示刷新或返回详情查看最新状态。
- 图片上传或保存失败时，编辑表单保留用户输入并展示失败原因。
- 访问不存在或无权访问的工单时，继续展示阶段 2 的未找到或无权限状态。

## 静态检查

交付前运行：

```powershell
npm run check
```

预期命令通过。若失败，必须先修复 TypeScript、Biome format 或 lint 问题；若环境原因无法运行，交付说明必须记录原因和风险。
