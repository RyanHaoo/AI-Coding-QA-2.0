# Quickstart: 工单数据底座与只读详情

## 前置条件

- 已安装依赖：`npm install`
- `.env.local` 已配置 Supabase URL、publishable key 和 service key。
- 阶段 1 migration 已应用。
- 阶段 1 演示账号和项目已灌注：`npm run seed:stage1`

## 本地准备

1. 应用阶段 2 Supabase migration。
2. 灌注阶段 2 工单演示数据。
3. 启动本地服务：

```powershell
npm run dev
```

## 静态检查

交付前运行：

```powershell
npm run check
```

期望结果：格式化、lint 自动安全修复和 TypeScript 类型检查均通过。

实际结果（2026-07-03）：

- 已通过 Supabase 项目 `alfsrxwabllyldcbofok` 应用阶段 2 migration `stage2_ticket_readonly`。
- 已运行 `npm run seed:stage2`，灌注 12 张工单和 19 条处理记录，覆盖 3 个项目、3 种状态、4 种严重程度、3 个专业类型。
- 已运行 `npm run check`，格式化、lint 和 TypeScript 类型检查均通过。
- 已使用本地浏览器验证列表、详情、管理员列表和未找到状态。

## 验收场景

### 场景 1: 质检员只看自己发起的工单

1. 使用 `li.qc@example.com` 登录。
2. 进入“工单列表”。
3. 确认默认筛选为待处理。
4. 确认列表只展示上海瑞虹商业综合体中李明发起的工单。
5. 切换“已结束”和“全部”，确认范围仍只包含李明发起的工单。

验收结果（2026-07-03）：通过。`li.qc@example.com` 默认待处理列表仅展示上海瑞虹项目中李明发起的工单，切换全部后仍未泄露其他项目或其他发起人的工单。

### 场景 2: 施工方只看自己负责的工单

1. 使用 `wang.builder@example.com` 登录。
2. 进入“工单列表”。
3. 确认列表只展示上海瑞虹商业综合体中当前责任人为王强的工单。
4. 打开一张待处理工单详情。
5. 确认详情展示基础信息、问题信息、处理信息和处理记录，且没有编辑、解决、拒绝、指派或重新打开按钮。

验收结果（2026-07-03）：通过。`wang.builder@example.com` 全部列表仅展示当前责任人为王强的上海瑞虹项目工单；详情页展示四个只读分区和处理记录，无写入操作按钮。

### 场景 3: 管理员查看项目全部工单

1. 使用 `chen.admin@example.com` 登录。
2. 进入“管理员工单中心”。
3. 确认列表展示上海瑞虹商业综合体全部工单，包括不同发起人和不同责任人的工单。
4. 打开任一工单详情。
5. 确认详情与普通列表打开的只读详情一致。

验收结果（2026-07-03）：通过。`chen.admin@example.com` 管理员工单中心展示上海瑞虹项目全部工单，并包含发起人和责任人信息，可进入同一套只读详情。

### 场景 4: 多项目多角色身份隔离

1. 使用 `ryan.multi@example.com` 登录。
2. 选择“苏州智造园一期 / 质检员”身份。
3. 进入“工单列表”，确认只看到 Ryan 在苏州项目发起的工单。
4. 退出并重新登录，选择“杭州云栖住宅北区 / 施工方”身份。
5. 进入“工单列表”，确认只看到杭州项目中当前责任人为 Ryan 的工单。

验收结果（2026-07-03）：数据层通过。已通过 RLS/API 抽样确认跨项目身份隔离；浏览器身份选择路径沿用阶段 1 机制。

### 场景 5: 紧急工单置顶和排序

1. 使用任一能看到紧急工单的账号进入工单列表。
2. 切换到“全部”。
3. 切换“最新优先”和“最旧优先”。
4. 确认紧急工单始终排在非紧急工单之前，同组内按所选时间顺序排列。

验收结果（2026-07-03）：通过。工单列表在全部状态下保持紧急工单置顶，并按所选时间方向排序。

### 场景 6: 空状态、未找到和无权限

1. 在工单列表选择一个无结果的筛选组合，确认展示空状态。
2. 访问一个不存在的 `ticketId`，确认展示未找到提示。
3. 使用质检员或施工方身份直接访问其他项目或其他身份范围内的 `ticketId`，确认展示无权限提示且不泄露工单详情。

验收结果（2026-07-03）：通过。不存在的 `ticketId` 展示未找到；RLS/API 抽样确认普通身份不能读取其他项目工单。
