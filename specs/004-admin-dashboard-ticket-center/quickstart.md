# Quickstart: 管理员态势大盘与工单中心

## 前置条件

- 已完成阶段 1、阶段 2 和阶段 3。
- `.env.local` 已配置 Supabase URL、publishable key 和服务端 `SECRET_KEY`。
- 阶段 1 测试账号可登录。
- 阶段 2/3 工单数据中至少包含当前项目的待处理、已完成、已拒绝和紧急工单。
- 阶段 3 管理员详情操作已可用。

## 启动

```powershell
npm run dev
```

本次实现验收使用本地开发地址 `http://localhost:3000`。建议优先使用：

- `chen.admin@example.com`: 上海瑞虹商业综合体 / 管理员，验证单项目管理员路径。
- `ryan.multi@example.com`: 多项目多身份，验证不同项目身份边界。
- `li.qc@example.com`: 质检员，验证非管理员拒绝访问。
- `wang.builder@example.com`: 施工方，验证非管理员拒绝访问。

## 验收路径 1: 查看管理员数据大盘

1. 使用 `chen.admin@example.com` 登录并选择管理员身份。
2. 打开 `/?view=dashboard` 或点击导航中的“数据大盘”。
3. 预期页面展示当前项目信息、工单总数、待处理、已完成、已拒绝和紧急工单数量。
4. 预期状态概览展示待处理、已完成、已拒绝分布。
5. 预期严重程度分布展示轻微、一般、严重、紧急分布。
6. 若当前项目存在紧急工单，预期重点关注区域明确提示；若没有，预期展示当前无紧急工单。

## 验收路径 2: 从大盘指标自动下钻

1. 在数据大盘点击“待处理”指标。
2. 预期进入 `/?view=admin-tickets`，且工单中心自动应用待处理状态筛选。
3. 预期筛选区可见“待处理”已生效，表格只展示待处理工单。
4. 返回数据大盘，点击“紧急”指标或重点关注入口。
5. 预期进入工单中心并自动应用紧急严重程度筛选。
6. 预期表格只展示紧急工单，且紧急工单有明确突出样式。

## 验收路径 3: 管理员工单中心筛选

1. 打开 `/?view=admin-tickets`。
2. 确认默认不预设状态筛选，展示当前项目全部工单或清晰的全部范围。
3. 分别按状态、严重程度、专业类型筛选，预期表格结果与筛选条件一致。
4. 输入问题关键词或详细位置关键词，预期结果匹配问题描述、详细位置或问题详情。
5. 输入完整或可识别工单编号，预期优先展示编号匹配工单。
6. 组合多个筛选条件，预期结果取交集。
7. 点击清空筛选，预期恢复当前项目全部范围。
8. 输入一个不存在的关键词，预期展示筛选无结果提示，并保留清空筛选入口。

## 验收路径 4: 表格打开详情并执行管理操作

1. 在管理员工单中心点击一张待处理工单的查看详情入口。
2. 预期展示完整工单详情、处理信息和处理记录。
3. 执行一项阶段 3 管理操作，例如解决、拒绝、编辑或指派他人。
4. 预期操作成功后新增处理记录。
5. 返回工单中心，预期表格中的状态、责任人或问题摘要反映最新结果。
6. 打开数据大盘，预期相关统计数量与最新状态一致。
7. 打开已完成或已拒绝工单，执行重新打开，预期状态恢复待处理，责任人保持不变，大盘和表格同步更新。

## 验收路径 5: 非管理员访问限制

1. 使用 `li.qc@example.com` 登录。
2. 直接访问 `/?view=dashboard` 和 `/?view=admin-tickets`。
3. 预期展示无权限反馈，不展示管理员大盘或当前项目全部工单。
4. 使用 `wang.builder@example.com` 重复以上步骤。
5. 预期同样展示无权限反馈。

## 验收路径 6: 多项目身份边界

1. 使用 `ryan.multi@example.com` 登录。
2. 选择上海瑞虹商业综合体 / 管理员身份。
3. 打开数据大盘和管理员工单中心，记录项目名称和工单范围。
4. 退出并重新登录，选择其他项目的非管理员身份。
5. 直接访问管理员视图，预期无权限。
6. 若后续存在其他项目管理员身份，切换后预期大盘和工单中心只展示该项目数据。

## 异常验收

- 当前项目没有任何工单时，大盘所有指标显示 0，工单中心展示空状态。
- 筛选条件无结果时，展示空结果说明和清空筛选入口。
- 直接访问不存在的 `ticketId` 时，展示未找到反馈。
- 访问不属于当前项目的 `ticketId` 时，展示无权限反馈。
- 管理员打开旧详情后若工单状态已变化，提交不再允许的阶段 3 操作时，系统拒绝并提示查看最新状态。

## 静态检查

交付前运行：

```powershell
npm run check
```

预期命令通过。若失败，必须先修复 TypeScript、Biome format 或 lint 问题；若环境原因无法运行，交付说明必须记录原因和风险。
